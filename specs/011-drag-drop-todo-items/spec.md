# Feature Specification: 拖拉式任務管理 (Drag-and-Drop Task Management)

**Feature Branch**: `011-drag-drop-todo-items`  
**Created**: 2026-05-11  
**Status**: Draft  
**Input**: User description: "新增一項目求, 我要將 TODO list 中的每一個 item 都能拖拉, 例如將 '進行中' 的 item 拉進 '測試中'的河道, 依此需求建立新的 branch"

## Clarifications

### Session 2026-05-11
- Q: 當多位使用者同時嘗試移動同一個任務卡片時，系統應如何處理衝突？ → A: 最後一次操作勝出 (Last Write Wins)。
- Q: 系統應該在何時向後端發送更新請求？ → A: 放下時立即更新 (On Drop)。
- Q: 任務在資料庫中如何維持排序？ → A: 權重欄位 (Rank field)：為每個任務維護一個排序值，移動時僅更新受影響的單一任務。
- Q: 系統是否需要提供「復原 (Undo)」功能，或者僅靠再次拖拉來修正？ → A: 再次拖拉修正 (Manual Correction)：不提供專門的復原按鈕，使用者若放錯位置，手動將其拖回即可。
- Q: 系統是否需要支援拖曳時的「自動捲動」？ → A: 支援自動捲動 (Auto-scroll)：當拖曳卡片靠近河道頂部或底部邊緣時，河道內容自動向上或向下捲動。

## User Scenarios & Testing *(mandatory)*


### User Story 1 - 任務狀態跨河道更新 (Priority: P1)

身為一個使用者，我想要能夠直接將任務卡片從一個狀態河道（例如「進行中」）拖曳到另一個河道（例如「測試中」），以便快速且直觀地更新任務進度。

**Why this priority**: 這是本需求的核心功能，直接解決了使用者想要透過拖拉操作來管理任務進度的痛點。

**Independent Test**: 使用者開啟任務列表，選擇一個任務，將其從 A 河道拖曳至 B 河道並放下，驗證任務卡片是否停留在 B 河道且狀態已更新。

**Acceptance Scenarios**:

1. **Given** 任務列表顯示「待辦」與「進行中」河道，**When** 使用者將一個「待辦」任務拖入「進行中」河道，**Then** 任務應該成功轉移至「進行中」河道。
2. **Given** 任務已成功拖入新河道，**When** 重新整理頁面，**Then** 該任務仍應出現在新河道中。

---

### User Story 2 - 拖拉過程中的視覺提示 (Priority: P2)

身為一個使用者，在拖曳任務時我希望能看到明確的視覺回饋（例如：卡片半透明、目標位置出現佔位框），這樣我才能清楚知道任務將被放置在哪裡。

**Why this priority**: 視覺回饋對於提升使用者體驗（UX）至關重要，可以防止誤操作並增加操作的確定感。

**Independent Test**: 當使用者開始拖曳卡片時，觀察卡片外觀是否有變化，以及當卡片懸浮在不同河道上方時，是否有對應的提示（如：河道背景變色或出現虛線框）。

**Acceptance Scenarios**:

1. **Given** 使用者正在拖曳任務卡片，**When** 卡片被提起時，**Then** 被拖曳的卡片應顯示為半透明。
2. **Given** 使用者將卡片懸浮在某個河道上方，**When** 該河道可接受此任務，**Then** 該河道應顯示明顯的提示（例如邊框加深）。

---

### User Story 3 - 河道內任務排序 (Priority: P3)

身為一個使用者，我想要在同一個河道內透過拖拉來調整任務的上下順序，以便根據重要性排列任務。

**Why this priority**: 雖然使用者主要要求的是跨河道的拖拉，但在同一個河道內排序是拖拉操作中常見的預期行為。

**Independent Test**: 在同一個河道內，將下方的任務拖曳到上方任務之前，驗證兩者位置是否互換。

**Acceptance Scenarios**:

1. **Given** 同一河道內有多個任務，**When** 使用者將任務 A 拖到任務 B 之上並放下，**Then** 任務 A 應該顯示在任務 B 的上方。

---

### Edge Cases

- **拖曳至無效區域**: 當使用者將任務拖曳到河道以外的空白區域或不允許放置的區域時，卡片應平滑地回到原始位置。
- **拖曳中連線中斷**: 若在拖曳並放下後，與伺服器的同步失敗，系統應提示錯誤並將卡片恢復到原始狀態。
- **空河道拖入**: 能夠成功將任務拖入一個目前沒有任何任務的空河道中。
- **多人同時操作衝突**: 當多位使用者同時移動同一個任務時，系統採用「最後一次操作勝出 (Last Write Wins)」策略，後到的請求將覆蓋先前的狀態。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系統必須支援使用滑鼠（Desktop）進行任務卡片的拖曳操作。
- **FR-002**: 系統必須在拖曳過程中動態顯示佔位符（Placeholder），以指示放下後的位置。
- **FR-003**: 系統必須在任務放下（Drop）後，自動偵測目標河道並更新任務狀態。
- **FR-004**: 系統必須在任務放下（Drop）時立即透過後端 API 持久化任務的新狀態與排序。
- **FR-005**: 系統必須支援跨河道移動以及同河道內重排。
- **FR-006**: 系統必須支援拖曳時的自動捲動 (Auto-scroll)，當卡片靠近河道邊緣時自動觸發。

### Key Entities

- **Task Item**: 代表可被拖曳的任務卡片，包含 ID、標題、描述、所屬狀態及 **Rank (排序權重)** 等屬性。
- **Status Lane (河道)**: 代表不同的任務階段（如：待辦、進行中、已完成），作為任務拖放的目標容器。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 使用者完成一次跨河道拖放操作（從按住到放開）應在 2 秒內完成視覺回應。
- **SC-002**: 拖拉操作引起的狀態更新成功率應達到 100%（在網路正常情況下）。
- **SC-003**: 在最新版本的 Chrome, Firefox, Safari 瀏覽器中，拖拉互動應保持流暢（無明顯卡頓）。

## Assumptions

- 假設目前的系統前端具備足夠的 DOM 結構來區分不同的狀態河道。
- 假設現有的任務 API 支援部分更新（Patch）任務狀態的功能。
- 暫不支援行動裝置（觸控）的拖拉優化，以 Desktop Web 為優先。
