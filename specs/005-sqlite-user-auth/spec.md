## User Scenarios & Testing *(mandatory)*

### User Story 1 - 安全的身份驗證 (Priority: P1)

作為一名使用者，我希望能夠使用我的憑據登入系統，以便存取我的個人資料並讓系統識別我的身份。

**Why this priority**: 這是任何受限存取系統的核心功能，對安全性和使用者識別至關重要。

**Independent Test**: 可透過嘗試使用有效與無效的憑據登入來測試。成功標準為正確識別身份並啟動會話 (Session)。

**Acceptance Scenarios**:

1. **Given** 已存在註冊使用者, **When** 提供正確的使用者名稱與密碼, **Then** 系統授予存取權限並導向主頁面。
2. **Given** 未註冊使用者或錯誤憑據, **When** 嘗試登入, **Then** 系統拒絕存取並顯示清晰的錯誤訊息。

---

### User Story 2 - 基於角色的存取控制 (RBAC) (Priority: P2)

作為一名管理員，我希望能夠定義並分配權限給使用者，以便控制誰可以存取特定功能（例如：編輯 vs. 僅查看）。

**Why this priority**: 對於管理多個具有不同職責的使用者至關重要，確保資料完整性。

**Independent Test**: 可透過使用不同權限的使用者帳號登入，驗證功能是否根據其分配的角色正確啟用或停用。

**Acceptance Scenarios**:

1. **Given** 具有「檢視者 (Viewer)」權限的使用者, **When** 嘗試刪除或編輯任務, **Then** 系統應封鎖該動作。
2. **Given** 具有「管理員 (Admin)」權限的使用者, **When** 進行修改, **Then** 系統允許並持久化該變更。

---

### User Story 3 - 使用者註冊 (Priority: P3)

作為一名新使用者，我希望能夠建立帳號，以便開始使用系統。

**Why this priority**: 這是擴展使用者群的必要流程。

**Independent Test**: 可透過填寫註冊表單並驗證資料庫中是否建立了新記錄，且該使用者隨後能成功登入來測試。

**Acceptance Scenarios**:

1. **Given** 註冊頁面, **When** 提供有效的詳細資訊, **Then** 系統建立新的使用者帳號。
2. **Given** 已存在的使用者名稱, **When** 嘗試註冊, **Then** 系統應防止重複建立帳號並提示。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系統必須將使用者資料（使用者名稱、雜湊後的密碼及權限）儲存在 SQLite 資料庫中。
- **FR-002**: 系統必須提供安全的登入介面，並根據資料庫驗證憑據。
- **FR-003**: 系統必須在成功登入後維護使用者會話狀態。
- **FR-004**: 系統必須支援多使用者網頁環境，處理伺服器端會話 (Session) 或 JWT，並確保網路傳輸安全。
- **FR-005**: 系統必須支援至少兩個不同的權限級別（例如：使用者與管理員）。
- **FR-006**: 系統必須控管所有 CRUD 操作：具有「檢視者」權限的使用者僅能查看任務，而「編輯者」或「管理員」則可執行建立、更新與刪除動作。
- **FR-007**: 系統必須提供登出功能，透過在用戶端清除存儲的身份驗證 Token (JWT) 來結束會話。
- **FR-008**: 系統必須允許具有「管理員」權限的使用者為其他使用者手動重設密碼。
- **FR-009**: 身份驗證 Token (JWT) 的有效期必須設定為 24 小時，過期後使用者必須重新登入。
- **FR-010**: 系統初次執行時，必須自動建立一個預設的管理員帳號 (username: admin, password: admin)，以便進行初始權限管理。
- **FR-011**: 系統必須提供專屬的管理員儀表板頁面，僅限具有「管理員」權限的使用者存取，用於管理使用者帳號及權限。
- **FR-012**: 系統必須實作嚴格的資料隔離，確保使用者僅能存取、建立、更新或刪除屬於其帳號的任務資料。
- **FR-013**: 系統必須具備暴力破解防護機制：若帳號連續登入失敗達 5 次，系統應暫時鎖定該帳號 15 分鐘，期間拒絕所有登入嘗試。

### Key Entities *(include if feature involves data)*

- **User (使用者)**: 代表具備系統存取權限的個人。屬性：ID、使用者名稱、密碼（雜湊）、角色 ID。
- **Role/Permission (角色/權限)**: 定義使用者的權限範疇。屬性：角色 ID、角色名稱、權限旗標。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 登入程序（從點擊「登入」到成功導向）在正常負載下應少於 1 秒。
- **SC-002**: 100% 的失敗登入嘗試應被記錄，以便進行安全性稽核（需遵守 Logger 規範）。
- **SC-003**: 密碼嚴禁以明文儲存，100% 必須使用符合業界標準的雜湊算法（如 bcrypt）。
- **SC-004**: 使用者無法透過直接操作 URL 或 API 繞過權限檢查。

## Assumptions

- 根據使用者要求，採用 SQLite 作為持久化存儲媒介。
- 系統將遵循專案憲章，將業務邏輯封裝於 Service 層。
- 密碼加密將採用業界標準實務（如 bcrypt 或 Argon2）。
- 假設目前應用程式尚無身份驗證層，此功能將作為基礎架構。
